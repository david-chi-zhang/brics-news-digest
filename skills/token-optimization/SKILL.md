# Token 节约最佳实践

## 核心理念
**Text > Brain 📝** 同样适用于 token 管理：**API > Browser > Manual**

## 一、工具选择优先级

| 优先级 | 工具 | 适用场景 | Token 消耗 |
|--------|------|----------|-----------|
| **1️⃣** | `web_fetch` | 提取网页可读内容 | ⭐ 低 |
| **2️⃣** | `browser` | 需要交互（点击、填表、截图） | ⭐⭐⭐ 高 |
| **3️⃣** | `feishu_*` API | 飞书操作 | ⭐ 低 |

**原则：** 能用 `web_fetch` 绝不用 `browser`。browser 的 snapshot 返回完整 DOM 结构，token 消耗是 web_fetch 的 5-10 倍。

## 二、网页内容采集优化

### ❌ 错误做法
```
1. browser.open 打开页面
2. browser.snapshot 获取页面结构
3. 分析 DOM 找链接
4. 再 fetch 每个链接
```

### ✅ 正确做法
```
1. web_fetch 列表页（extractMode=markdown）
2. 从 markdown 中提取链接和标题
3. 用标题/摘要预过滤相关性
4. 只 fetch 相关的文章全文
```

### 进阶技巧

**1. 限制返回长度**
```yaml
web_fetch:
  maxChars: 5000  # 列表页不需要全文
```

**2. 两阶段过滤**
```
阶段 1: fetch 列表页 → 提取标题 + 链接 → 过滤相关性
阶段 2: 只 fetch 相关文章全文
```

**3. 摘要模式**
```yaml
# 只需要摘要时
web_fetch:
  maxChars: 1000  # 限制每篇文章返回量
  extractMode: text  # 比 markdown 更精简
```

## 三、并行处理

**独立任务必须并行**，不能串行：

```yaml
# ✅ 好的：同时发出多个 fetch
同时 fetch 8 篇文章

# ❌ 坏的：串行
fetch 文章 1 → 等完成 → fetch 文章 2 → ...
```

## 四、飞书文档优化

### 内容精简
- 只保留必要信息（标题、链接、50 字摘要）
- 避免重复内容
- 用表格代替长段落

### 文档结构
```markdown
# 标题

## 分类 1
- 文章 1：标题 + 链接 + 摘要
- 文章 2：标题 + 链接 + 摘要

## 关键数据摘要（表格）
| 指标 | 数值 | 趋势 |

## 主题总结（3-5 点）
```

## 五、检查清单

执行网页采集任务前，快速检查：

- [ ] 是否可以用 `web_fetch` 代替 `browser`？
- [ ] 列表页是否设置了 `maxChars` 限制？
- [ ] 是否在 fetch 全文前先用标题过滤？
- [ ] 独立任务是否并行处理？
- [ ] 输出是否精简（只保留必要信息）？
- [ ] 是否避免了重复 fetch 相同 URL？

## 六、典型场景对比

### 场景：采集新闻摘要

| 步骤 | ❌ 高消耗 | ✅ 低消耗 |
|------|----------|----------|
| 1 | browser.open + snapshot | web_fetch 列表页 |
| 2 | 分析 DOM 找链接 | markdown 中提取链接 |
| 3 | fetch 所有文章全文 | 标题过滤后 fetch 相关 |
| 4 | 生成详细摘要 | 50 字精简摘要 |
| 5 | 长文档输出 | 结构化精简文档 |

**Token 节省：约 60-80%**

## 七、特殊情况

**必须用 browser 的场景：**
- 需要登录才能看到内容
- 需要点击/翻页/填表
- 需要截图
- 动态加载内容（web_fetch 无法获取）

**其他情况优先用 web_fetch。**
